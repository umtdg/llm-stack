View 1 / 3 — logical

Layers and the protocols connecting them

Three independent consumers sit on top: the chat client, the eval bench, and the knowledge base. Chat and eval both consume the knowledge base for retrieval and the model gateway for generation; the knowledge base is itself a consumer of the gateway. The gateway is the single fan-out point to every model backend.

Layer roles

  • Chat client / agent — the interactive surface. Captures input, renders, runs the agent loop. Calls the gateway for generation and the knowledge base for retrieval; owns a private conversation store.
  • Eval bench — offline scoring. Calls the knowledge base to score the live retrieval pipeline and the gateway for query embeddings and judge-LLM calls; owns a private results store.
  • Knowledge base (RAG service) — owns ingestion and retrieval and the four backing stores. The only layer that touches the stores; everything above sees only its retrieval API.
  • Model gateway / router — one OpenAI-compatible endpoint in front of all backends. Routing, key custody, budgets, fallback. The single egress chokepoint for model traffic.
  • Self-hosted engines — local inference and embedding serving.
  • Proprietary providers — external first-party / cloud model APIs reached by key.

Every connection, by protocol family

A Model calls — OpenAI-compatible /v1 over HTTP, streamed with SSE

  • Chat client → gateway: chat completions for generation and the agent loop.
  • Eval bench → gateway: query embeddings plus judge-LLM calls.
  • Knowledge base → gateway: ingest embeddings on /embeddings, entity/relation extraction on /chat/completions, and query-time embeddings.
  • Gateway → self-hosted engines: /v1 over the LAN (loopback if co-located).
  • Gateway → proprietary providers: the provider's HTTPS API (native or an OpenAI-compat shim) over the WAN.

B Knowledge access — MCP or a retrieval HTTP API

  • Chat client → knowledge base: retrieval; request/response returning ranked chunks plus citations.
  • Eval bench → knowledge base: same edge, used to score retrieval quality.

C Datastore access

  • Internal to the knowledge base — detailed in View 2.

Control / admin and secrets (off the data path)

A control plane feeds the gateway's routing/registry table and drives self-hosted engine lifecycle (pull / deploy / health) over an admin HTTP or orchestration API. The gateway reads provider keys from a secret store (OS keychain or secret manager) via a local credential read. Each consumer also owns a private store off the shared path — the chat client a conversation store, the eval bench a results store — typically SQL or embedded.

View 2 / 3 — knowledge base internals

The ingest and query paths, and where the stores live

Two flows run through one service. Ingest writes; query reads. The knowledge base is the only component that touches the vector, graph, object, and metadata stores.

Ingest writes

A document arrives; the service chunks it, calls the gateway for chunk embeddings and writes vectors to the vector store, calls the gateway for entity/relation extraction and writes nodes and edges to the graph store, stores the raw file and chunk text in the object store, and records collection and chunk metadata — including which embedding model the collection is bound to — in the metadata DB.

Query reads

A query arrives; the service embeds it via the gateway, runs ANN search against the vector store and a neighborhood traversal against the graph store, fuses and ranks the two result sets, hydrates chunk text from the object store, and returns ranked context over the retrieval API (or MCP).

Store roles vs protocols

  • Vector store — reached through an ANN driver: an in-process library call when embedded, gRPC/HTTP when it's a server.
  • Graph store — a graph query: embedded edges, a Bolt/Cypher session, or SQL.
  • Object store — S3-compatible API or a filesystem path.
  • Metadata DB — SQL, embedded or networked.

The embedder is per-collection metadata, so a collection's vector index has a fixed dimension; you generally keep one index per collection rather than one shared table.

View 3 / 3 — deployment

Network topology and where each component can run

The only component that must be remote is the proprietary providers. Everything else can be fully local, and the gateway is the single egress for all model traffic.

Per-component placement

  • Chat client — desktop app (everything on one host) or a workstation talking to a LAN gateway and KB.
  • Knowledge base service — same host as the client, or a shared LAN service. Stores embed in-process (one binary, zero network) or split onto a DB node.
  • Eval bench — same host, a separate dev box, or CI. Offline, so never needs co-location.
  • Model gateway — co-located (loopback), a LAN GPU node, or a hosted instance. Routing chokepoint regardless of placement.
  • Self-hosted engines — same host (small models) or a GPU node on the LAN.
  • Proprietary providers — always WAN/HTTPS. The only forced-remote component.
  • Vector / graph / object / metadata stores — each independently embedded (in-process, same host) or a networked service (LAN or managed cloud).

The gateway is the only path to external providers, so it is where egress policy is enforced. The data planes that hold the corpus (KB + stores) can stay entirely on-prem even while generation uses a proprietary model. Per-collection policy decides whether a collection's content is ever allowed across that boundary.

Consequence

Build vs adopt

Every box and every store in these views is a role with an off-the-shelf implementation and a standard protocol. Adopt all of them: engines, gateway/router, vector store, graph store, object store, metadata DB, the RAG/KB framework, the eval framework, the MCP server, the chat shell. Building any of them is rebuilding a solved layer.

What does not exist as a single off-the-shelf product is the wiring itself: the control plane that binds a collection to an embedder, routes per-chat and per-message to a collection and a model, enforces the egress policy at the gateway, and drives the eval → fine-tune → redeploy loop end to end. That orchestration spans four layers and is owned by none of them. That — plus the product's UX — is the entire custom surface.

The test: if what you build is that glue, it makes sense; if it starts reimplementing a layer in these diagrams, it does not.

Knowledge base the RAG framework that owns ingest + retrieval

Two shapes hide here, and the difference decides how much you build. A deployable service already exposes a retrieval API/MCP — it is the KB box. A library is code you import; to become the KB box it needs a service shell, a retrieval API/MCP surface, and any pipeline stage the library leaves out. The architecture's KB box is service-shaped, so favour the service rows unless you have a reason to own the pipeline.

ToolShapeBest forIntegration & trade-off
LightRAG ★service (+SDK)graph-RAG that drops straight into every other layerthe lightrag-server ships a REST API, a community MCP bridge, and an Ollama-compatible endpoint the chat shell calls as if it were a model. Consumes OpenAI-compatible LLM/embeddings, so it sits behind LiteLLM; pluggable backends (Postgres, Neo4j, Qdrant, Milvus, Redis, Faiss…) become the vector/graph stores of Fig. 2. PickLightRAG — it is the rare KB that already speaks all three integration directions: REST/MCP/Ollama out to the chat shell and eval bench, OpenAI-compatible in to the gateway, pluggable drivers down to the stores. That is the "seamless integration" the rest of the diagram assumes, with near-zero glue. Use R2R if you don't want a graph layer. Drop to a library (LlamaIndex) only when you need control the services won't give you — and budget for the service shell + extraction pipeline you'll then own. For extraction quality beyond LightRAG: Microsoft GraphRAG (hierarchical communities, costly to re-index) or Graphiti (temporal / agent-memory graphs).

Cross-cutting framework vs service — what you hand-build

A framework/library only becomes a layer once it runs as a process and speaks one of the three protocol families to its neighbours. Picking a library is choosing to build that adapter — and any pipeline stage the library omits. This is the hidden cost behind "just use LlamaIndex." The deployable services above already pay it.

RuleAsk of every candidate: does it ship an API surface, or is it a library I import? A library means you own a long-lived service, its protocol adapter, and the pipeline gaps — often more work than the layer it's "providing." Eval frameworks are the exception: the bench is meant to be code you run, so DeepEval/Ragas-as-libraries is the correct shape, not a gap.

Chat client / agent the interactive surface

A frontend, not an engine — it talks to the gateway for generation and the KB for retrieval. Two axes decide it: local-model polish vs. multi-provider breadth, and — your requirement — how it takes plugins. The portable answer to extensibility is MCP: build an extension once as an MCP tool server and every shell below that speaks MCP can load it, so you avoid one shell's proprietary plugin format.

Reference stacks

Two coherent end-to-end picks from the tables above, at the two deployment poles.

  • Single-host / embedded — Ollama · LiteLLM · LanceDB · Kuzu · LightRAG-server (graph-RAG, REST+MCP+Ollama-compat in one binary) · filesystem + SQLite · Ragas + Langfuse · Open WebUI with MCP-authored plugins. One trust boundary, near-zero networked dependencies, runs on a workstation.
  • Shared LAN service — vLLM/SGLang · LiteLLM · Qdrant · Neo4j · LightRAG-server (Postgres/Qdrant/Neo4j drivers) or a LlamaIndex-built service if you need the control · MinIO + PostgreSQL · DeepEval + Phoenix · LibreChat with MCP tools. Scales to a team and multi-tenant traffic; the gateway is still the only egress.

Every tool above implements a role already drawn in Figs. 1–3 and speaks one of the three protocol families. None of them is the control plane that binds collection→embedder, routes per-message to a collection+model, enforces egress at the gateway, and drives the eval loop — that glue remains the custom surface from "Build vs adopt." Adopt the rows; build only the wiring.
